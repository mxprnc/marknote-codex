# 모델, 설정, 권한, 도구 연동

이 문서는 Codex CLI의 모델 선택, 기능 플래그, 승인 모드, 웹 검색, MCP, 서브에이전트 설정을 정리합니다.

## 모델과 reasoning

대부분의 Codex 작업에는 사용 가능할 경우 `gpt-5.5`가 권장 모델입니다. 복잡한 코딩, 컴퓨터 사용, 지식 작업, 리서치 워크플로에 적합하며, 다단계 작업에서 계획 수립, 도구 사용, 후속 처리 능력이 강화된 frontier 모델로 설명됩니다.

`gpt-5.5`를 아직 사용할 수 없다면 `gpt-5.4`를 계속 사용할 수 있습니다. 빠른 작업이 필요하고 ChatGPT Pro 구독자라면 research preview로 제공되는 GPT-5.3-Codex-Spark 모델을 사용할 수 있습니다.

CLI 시작 시 모델 지정:

```bash
codex --model gpt-5.5
```

세션 중에는 `/model` 명령으로 모델을 바꿀 수 있습니다.

```text
/model
```

## 모델 선택 기준

| 작업 | 추천 방향 |
| --- | --- |
| 복잡한 리팩터링, 대규모 구현, 장시간 조사 | `gpt-5.5` |
| 일반적인 코딩, 리뷰, 문서 작업 | `gpt-5.5` 또는 `gpt-5.4` |
| 아주 빠른 응답이 중요한 작은 작업 | GPT-5.3-Codex-Spark 사용 가능 여부 확인 |
| 비용과 속도를 더 중시하는 반복 작업 | 낮은 reasoning 또는 빠른 모델 고려 |

## 기능 플래그

Codex에는 실험적이거나 점진적으로 배포되는 기능을 켜고 끌 수 있는 feature flag가 있습니다.

사용 가능한 기능 목록 확인:

```bash
codex features list
```

기능 켜기:

```bash
codex features enable unified_exec
```

기능 끄기:

```bash
codex features disable shell_snapshot
```

`enable`과 `disable`은 기본적으로 `~/.codex/config.toml`에 설정을 저장합니다. `--profile`로 Codex를 실행한 경우에는 root 설정이 아니라 해당 profile에 저장됩니다.

## 승인 모드

승인 모드는 Codex가 사용자 확인 없이 어디까지 수행할 수 있는지 정의합니다. 인터랙티브 세션 안에서는 `/permissions`로 변경할 수 있습니다.

```text
/permissions
```

### Auto

기본 모드입니다. Codex는 작업 디렉터리 안에서 파일을 읽고, 편집하고, 명령을 실행할 수 있습니다. 다만 범위를 벗어나는 파일 접근이나 네트워크 사용 등은 사용자 승인을 요청합니다.

일반적인 저장소 작업, 테스트 실행, 문서 수정에 적합합니다.

### Read-only

상담 모드에 가깝습니다. Codex가 파일을 탐색할 수는 있지만, 변경을 만들거나 명령을 실행하려면 사용자의 승인을 받아야 합니다.

다음 상황에 적합합니다.

- 코드베이스 설명만 받고 싶을 때
- 변경 전 설계 검토가 필요할 때
- 신뢰하기 어려운 저장소를 먼저 읽기만 할 때

### Full Access

Codex가 네트워크 접근을 포함해 머신 전체에서 작업할 수 있습니다. 강력하지만 위험도가 가장 높습니다.

신뢰할 수 있는 저장소와 명확한 작업에서만 제한적으로 사용하는 것이 좋습니다.

Codex는 항상 수행한 작업의 transcript를 보여주므로, 필요하면 일반적인 git workflow로 변경 사항을 검토하고 되돌릴 수 있습니다.

## 웹 검색

Codex CLI에는 first-party 웹 검색 도구가 포함됩니다. 로컬 작업에서는 기본적으로 웹 검색이 활성화되며, 검색 결과는 OpenAI가 유지하는 웹 검색 캐시에서 제공됩니다.

캐시 모드는 실시간 페이지를 직접 가져오는 대신 미리 인덱싱된 결과를 반환합니다. 임의의 라이브 웹 콘텐츠로부터 prompt injection에 노출되는 면을 줄이는 효과가 있지만, 웹 검색 결과는 여전히 신뢰할 수 없는 입력으로 취급해야 합니다.

최신 데이터가 필요한 경우 단일 실행에 `--search`를 넘기거나 설정에서 live 검색을 켤 수 있습니다.

```bash
codex --search "오늘 기준 Node.js LTS 버전 확인해줘"
```

설정 예:

```toml
web_search = "live"
```

웹 검색을 끄려면:

```toml
web_search = "disabled"
```

`--yolo` 또는 full access sandbox 설정을 사용하는 경우 웹 검색은 기본적으로 live 결과를 사용합니다.

Codex가 검색을 수행하면 transcript 또는 `codex exec --json` 출력에 `web_search` 항목이 나타납니다.

## 서브에이전트

서브에이전트는 큰 작업을 병렬화하기 위한 기능입니다. 예를 들어 하나의 에이전트가 백엔드 수정 범위를 조사하는 동안 다른 에이전트가 프론트엔드 영향 범위를 분석하게 할 수 있습니다.

중요한 특징은 다음과 같습니다.

- Codex는 사용자가 명시적으로 요청할 때만 서브에이전트를 생성합니다.
- 각 서브에이전트는 자체 모델 호출과 도구 사용을 수행합니다.
- 따라서 단일 에이전트로 처리하는 것보다 더 많은 토큰을 사용할 수 있습니다.

서브에이전트의 role 설정은 `config.toml`의 `[agents]` 섹션에서 구성합니다. 자세한 설정과 예시는 원문 Subagents 문서를 참고합니다.

개발자가 실무에서 사용할 때는 다음 기준이 좋습니다.

| 상황 | 서브에이전트 적합도 |
| --- | --- |
| 독립적인 여러 영역을 동시에 조사 | 높음 |
| 서로 다른 파일 영역을 병렬 구현 | 높음 |
| 한 파일의 섬세한 수정 | 낮음 |
| 다음 단계가 특정 조사 결과에 막혀 있음 | 낮음 |
| 토큰 비용보다 속도가 중요함 | 높음 |

## Model Context Protocol(MCP)

MCP는 Codex를 더 많은 도구와 연결하기 위한 프로토콜입니다. STDIO 또는 streaming HTTP 기반 MCP 서버를 `~/.codex/config.toml`에 추가하거나, `codex mcp` CLI 명령으로 관리할 수 있습니다.

Codex는 세션 시작 시 설정된 MCP 서버를 자동으로 실행하고, 해당 서버가 제공하는 도구를 built-in 도구 옆에 노출합니다.

예상 활용 사례:

- 사내 이슈 트래커 연결
- 데이터베이스 스키마 조회 도구 연결
- 문서 검색 서버 연결
- 빌드/배포 시스템 제어 도구 연결

Codex 자체를 MCP 서버로 실행해 다른 agent 안에서 사용할 수도 있습니다.

자세한 설정, 인증 흐름, 예제 구성은 원문 Model Context Protocol 문서를 참고합니다.
